### 5. Safe Sync & The Text Normalization Engine

**These two tests are the most valuable on the list and expose actual bugs in your current implementation.**

* **The UTF-16 Indexing Mismatch (CRITICAL):** The AI nailed this one. Python's `len(new_full_text)` counts characters (e.g., the emoji `📊` has a length of 1). However, the Google Docs API calculates indices based on **UTF-16 code units** (where `📊` has a length of 2). If your draft text contains emojis, complex unicode characters, or surrogate pairs, your `new_length` calculation will be lower than what Google Docs expects. The subsequent `deleteContentRange` will leave behind leftover characters, slowly corrupting the document.
* **The Invisible Edit Guard:** Another great catch. Your current `_normalize` function relies on `.split()` to collapse whitespace. Standard Python `.split()` does *not* treat zero-width spaces (`\u200b`) as whitespace. Therefore, if a zero-width space is introduced, your normalizer won't strip it, the strings won't match, and the `GDocsConflictError` will fire incorrectly.

### 1. The UTF-16 Length Helper

Python strings are inherently Unicode, and `len()` counts human-readable characters. Google Docs API counts UTF-16 code units. The cleanest way to calculate this in Python is to encode the string to UTF-16 Little Endian (which uses exactly 2 bytes per code unit) and divide the byte length by 2.

Drop this helper into `google_docs_service.py`:

```python
def _get_utf16_length(text: str) -> int:
    """
    Calculates the length of a string in UTF-16 code units.
    Essential for Google Docs API indexing, where surrogate pairs 
    (emojis, complex math symbols) count as 2 units instead of 1.
    """
    return len(text.encode('utf-16-le')) // 2

```
