Here is the final, strictly technical list of flaws and missing guards in your plan, based directly on the architecture of `notebooklm_service.py`. No filler.

1. Reuse the Exact JSON Cleaning Pattern
In notebooklm_service.py, inside suggest_summary_service, you have a highly battle-tested block of code for extracting JSON from NotebookLM's occasionally messy text responses:

### 1. Missing File Size Guard (The 10MB Limit)

* **The Flaw:** Your plan skips checking PDF file sizes before uploading.
* **What `nlm_service` does better:** It explicitly enforces a `PDF_SIZE_LIMIT_MB = 10.0` guard (lines 118-142) before touching the `add_drive` or `add_file` methods. NotebookLM historically chokes or times out on massive PDFs.
* **The Fix:** Before entering the upload loop in Step 7, run the files through a size check. If a file exceeds 10MB, halt the job for that thesis and set the status to `error: file_too_large`, rather than letting the upload semaphore hang and fail ungracefully.

### 2. Missing In-Memory Mutex (The "Double-Click" Race Condition)

* **The Flaw:** You rely entirely on writing `misc/source_index_{name}.json` to disk to mark a job as "running". In an async FastAPI environment, if a user double-clicks the "Run" button, two requests will read the disk state as "idle" before the first one has time to write "running". You will spawn two concurrent upload loops for the exact same folder.
* **What `nlm_service` does better:** It uses an in-memory `asyncio.Lock()` registry (`_run_locks`, lines 62-81) to instantly block concurrent execution for the same target, bypassing disk I/O latency entirely.
* **The Fix:** Add an `_index_locks: dict[str, asyncio.Lock]` at the top of your new `source_index_service.py`. Wrap the entire background task in `async with await _get_index_lock(thesis_name):`.

### 3. Duplicating Path & Link Resolution

* **The Flaw:** Step 7 of your data flow says: *"If drive_link exists → extract file_id..."*. You are planning to write custom logic to parse the scan dictionary and extract the Google Drive regex.
* **What `nlm_service` does better:** It already has a highly robust, fully tested `_resolve_absolute_paths()` function (lines 185-236). This function already handles fallback path matching, extracts the Drive file ID using the correct regex, **and** handles file deduplication via a `seen` set.
* **The Fix:** Do not write custom extraction logic in the new service. Import `_resolve_absolute_paths` from `notebooklm_service.py` and pass it your source list. It will hand you back a clean list of dictionaries containing exact `abs_path` and `drive_file_id` values ready for upload.

### 4. No "Partial Upload" Guard (Silent Data Loss)

* **The Flaw:** Your plan assumes that if you loop over 5 PDFs, 5 PDFs will upload. If 2 of them time out, your plan moves straight to Step 8 (sending the prompt). NotebookLM will generate an index card for 3 PDFs, and completely ignore the 2 that failed. `do_auto_import` will succeed, the UI will say "Indexed", and you will silently be missing data.
* **What `nlm_service` does better:** Lines 390-410 enforce a strict Completeness Check. If `len(uploaded) < len(resolvable)`, it halts the pipeline, refuses to send the prompt, and sets the status to `waiting_for_manual_upload`.
* **The Fix:** Between Step 7 (Uploads) and Step 8 (Prompt), add a strict length comparison. If any files failed to upload, update the `misc/source_index_{name}.json` state to an error and **return early**. Do not ask the LLM to generate a JSON card for an incomplete dataset.

Looking at this from a UX/edge case perspective, here are the issues I see:

## UX Problems

**The "Run All Unindexed" button is dangerous without a preview.** If you have 12 thesis folders and click this, you get 12 simultaneous notebook creations, 12 upload loops, and if 3 fail partway through, you have no clear picture of what succeeded and what didn't. The batch NLM flow in the existing code has this same problem but it matters less there because each subtopic is small. Here, each "job" is an entire thesis folder. The blast radius of a batch error is much larger.

Better: Show a "You're about to index N folders (X PDFs total)" confirmation modal before the batch fires. Single folders should still run immediately.

**The "Re-run" button has destructive ambiguity.** If a folder is already indexed and has a `group_id` in the source library, clicking Re-run will call `do_auto_import` again which creates a **new** SourceGroup with a new `group_id`. You now have two copies of Sharma_2003 in Card 03 with no indication which is current. The plan doesn't address deduplication on re-import at all.

Better: Re-run should either delete the old group first (with a warning), or update it in place. Given that `do_auto_import` doesn't support upsert, the safest path is: warn the user "This will create a duplicate — delete the old one from Card 03 first", and show the existing `group_id` link.

**The upload mode badge being read-only is correct, but the label is confusing.** `🔗 URL` vs `📁 Local` doesn't tell you *why* a folder is local. If Drive links aren't registered for a folder, the user doesn't know whether that's intentional or a setup mistake. They'll wonder "should I go fix this first before running?"

Better: Show `🔗 Drive` in green when links exist, and `⚠ Local only` in amber when they don't, with a tooltip "Drive links not registered — upload will use local paths (slower)". This makes the missing registration feel like a warning, not a neutral status.

**The `[↗ JSON]` button in the indexed row has no defined behavior.** The UI layout shows it but the plan never says what it opens. Does it show the raw JSON that was generated? Does it link to the source group in Card 03? This will be confusing without a clear answer.

Better: Make it navigate to the corresponding group in Card 03 by scrolling to `group-${group_id}` and opening it, rather than showing raw JSON. The user cares about the imported result, not the intermediate JSON.

## Edge Cases Not Addressed

**What happens if the NLM response is valid JSON but doesn't match the expected schema?** `do_auto_import` will call `SourceImport(**data)` which will succeed with defaults — `author: "Unknown Author"`, `title: "Untitled Work"`, `key_claims: ["No specific claims extracted."]`. The job status will show "Indexed" but the index cards will be garbage. There's no quality signal to the user.

Better: After `do_auto_import`, check `result["sources_created"]`. If it's 0, or if all chapters have default/empty `key_claims`, set status to a `warn` state — "Indexed but may be incomplete — check Card 03".

**The notebook reuse logic breaks on renamed thesis folders.** Step 5 says "verify + reuse" the notebook by `notebook_id` stored in the scan entry. But if the user renames the folder locally and rescans, the entry gets a new key and the `index_notebook_id` is lost. The service will create a duplicate notebook. This is low frequency but the result (orphaned notebooks in NotebookLM) is annoying.

**The 3-second inter-upload sleep isn't in the plan.** The existing `_run_sequence` has a 10-second post-upload indexing buffer (`await asyncio.sleep(10)`) specifically because NotebookLM needs time to process sources before the prompt is sent. Your plan mentions "3s sleep between uploads" (which mirrors the rate limit buffer) but doesn't include the post-upload indexing wait. If you send the prompt immediately after the last upload completes, NLM may generate the index card against partially-indexed sources.

**The per-thesis state file name collides if thesis names have special characters.** `misc/source_index_{safe_name}.json` uses the thesis folder name. But the scan key is the raw folder name (e.g. "Sharma, R. (2003)"). The `safe_name` sanitization isn't defined in the plan. You need the same `replace("/", "_").replace("\\", "_").replace(":", "_")` treatment that `save_index_card` uses, plus stripping spaces and commas, otherwise the misc key lookup will fail silently.

**There's no timeout on the NLM prompt step.** `_ask_with_retry` has retry logic for `TimeoutException` but the timeout itself is set by the underlying httpx client default, which can be very long. The source card prompt is asking NLM to summarize an entire thesis folder — that's a longer operation than a section prompt. If NLM hangs on this step, your background task hangs indefinitely holding the lock, and the UI shows "Running…" forever. You need an explicit `asyncio.wait_for(..., timeout=300.0)` wrapper around the `_ask_with_retry` call, same as the upload steps use `asyncio.wait_for(..., timeout=180.0)`.

**Stopping a running job isn't addressed.** The write section has a Stop button that calls `DELETE /notebooklm/notebook/{chapter}/{subtopic}`. Card 02b's table layout shows a `[Stop]` button for running rows but the plan has no endpoint for it. Without this, a stuck job can only be cleared by restarting the server.
