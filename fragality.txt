### Step 1: The Anchor File (Solving Folder-Level Fragility)

Stop using folder names to link Local Files, Google Drive, and your Database. Folders are just wrappers; their names shouldn't matter to the system.

**How it works:**

1. When you create a thesis in SPO, the backend generates a tiny, hidden JSON file called `.spo_project` containing a single UUID: `{"thesis_id": "t_1773"}`.
2. You drop this `.spo_project` file into your local thesis folder. You also drop a copy into your Google Drive folder.
3. **The Magic:** We rewrite `drive.py`. When `scan-local` runs, it doesn't care what the folder is named. It uses `rglob` to find `.spo_project`, reads the UUID, and registers that folder to `t_1773`. When `register-links` walks Google Drive, it searches for `.spo_project`, reads the UUID, and links the Drive folder to `t_1773`.

**The Result:** You can name your local folder "Messy Drafts" and your Drive folder "Official Thesis". The backend perfectly syncs them because they share the Anchor File. The `thesis_name` is no longer a primary key.

### Step 2: Auto-Generated CiteKeys (Solving File-Level Fragility)

Right now, `source_resolver.py` uses wild regex (like mapping the Roman numeral 'IV' to '4') just to guess which PDF the LLM meant when it said `"source_id": "Chapter 4"`. This is insane technical debt.

**How it works:**

1. When `POST /drive/scan-local` discovers your PDFs, SPO automatically assigns a strict, immutable **CiteKey** (like BibTeX) to every file.
* `08_chapter 2.pdf` → `[SRC-01]`
* `Sharma_2003.pdf` → `[SRC-02]`


2. The SPO backend stores this exact dictionary: `{"SRC-01": "08_chapter 2.pdf"}`.
3. **The Prompt Injection:** When you ask Claude to write the Chapterization JSON, you inject this generated library into the prompt:
> *"Here is the available source library. When assigning sources to subtopics, you MUST use the exact CiteKey: [SRC-01]: 08_chapter 2.pdf, [SRC-02]: Sharma_2003.pdf"*



**The Result:** Claude will effortlessly output `"source_ids": [{"source_id": "SRC-01"}]`.

## The Core Problem: String-Matching as a Foreign Key

The SPO architecture currently relies on **human-readable strings** (folder names, filenames, and semantic labels like "Chapter 2") to act as primary keys linking local storage, Google Drive, NotebookLM, and the Chapterization JSONs.

Because humans constantly rename files, fix typos, and use inconsistent casing, these links frequently shatter. This fragility spawned hundreds of lines of technical debt (complex regex in `source_resolver.py`) and manual intervention tools (`fix_source_ids.py`) just to keep the system glued together.

---

## The Discarded Solutions (And Why They Failed)

Before arriving at the optimal fix, we explored two other paths that looked good on paper but failed under the reality of your specific architecture:

### 1. Immutable Google Drive IDs

* **The Idea:** Replace all file naming logic by using Google Drive's native file IDs (e.g., `1A2B3C...`) as the absolute source of truth across the entire system.
* **Why it was discarded:** * **Broke Local Fallback:** It immediately destroyed your local-only workflow (`upload_method == "local"`), locking the system into the cloud.
* **LLM Hallucinations:** Asking Claude to accurately generate a chapterization JSON using 15-character random alphanumeric strings is highly risky.
* **Human Unreadability:** The Chapterization JSONs would become impossible to debug manually.



### 2. The UUID Semantic Router

* **The Idea:** Utilize the UUIDs already being generated by your `source_importer.py` (`spo_id: a1b2c3d4`) and inject them into Claude's prompt alongside the human label.
* **Why it was discarded:** While it solved file-level mapping, it completely ignored **folder-level fragility**. It still assumed the local thesis folder and the Drive thesis folder shared the exact same string name.

## The Ultimate Fix: Anchor Files + Strict CiteKeys

This solution entirely decouples system identity from human naming conventions. It addresses the fragility at both the folder layer and the file layer without overengineering or requiring a separate database server.

### Pillar 1: The Anchor File (Folder-Level Fix)

Instead of linking your local `D:\PhD\Drafts` folder to your `Drive/Official Thesis` folder by trying to match their names, you use a hidden anchor.

* A tiny `.spo_project` JSON file containing a unique `thesis_id` is placed in both the local directory and the Drive directory.
* The system scans for this file, reads the UUID, and instantly knows these two folders are the same project, regardless of what the user named them.

### Pillar 2: Auto-Generated CiteKeys (File-Level Fix)

Instead of using regex to guess if "08_chapter 2.pdf" is what Claude meant by "Chapter 2", the system assigns a strict, immutable tag (like BibTeX) to every discovered PDF.

* `08_chapter 2.pdf` becomes `[SRC-01]`.
* This mapping is injected directly into Claude's prompt. Claude uses `[SRC-01]` in the JSON, and the backend resolves it with a simple `O(1)` dictionary lookup.

## Exact Implementation Guide

Here is how to surgically implement this into your current codebase:

**1. Update `drive.py` (The Scanner)**

* Modify `scan_local` and `register_drive_links` to stop grouping by folder name.
* Instead, have them search for the `.spo_project` file.
* When found, read the `thesis_id`. All PDFs in that directory (and subdirectories) are registered to that ID.

**2. Generate the CiteKey Registry**

* During the local scan, as PDFs are discovered, assign them a sequential or hashed CiteKey (`SRC-01`, `SRC-02`).
* Save this flat mapping to your SQLite database (or state JSON):
`{"SRC-01": {"local_path": "...", "drive_id": "...", "filename": "08_chapter 2.pdf"}}`

**3. Update the LLM Prompt**

* When generating the Chapterization JSON, inject the registry into the prompt:
> *"Available Sources: [SRC-01]: 08_chapter 2.pdf. You MUST use the exact CiteKey (e.g., SRC-01) in the 'source_id' field."*



**4. Update `compiler.py**`

* When the compiler needs to upload files, it simply reads the `source_id` from the JSON (which is now `SRC-01`) and pulls the exact Drive ID or local path directly from the registry.
