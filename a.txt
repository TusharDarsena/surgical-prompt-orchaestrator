This is a robust foundation. Building a structured curation tool for academic writing requires strict guardrails, and your use of explicit Stage 1 (Generation) and Stage 2 (Scholarly Expansion) prompts ensures the output remains rigorous.

However, looking at the architecture of the orchestrator, the current implementation relies heavily on transient memory and brittle timing mechanisms. To scale this effectively, we need to move from a script-like execution model to a resilient, distributed architecture.

Here is a breakdown of major architectural upgrades across the three core areas you identified.

---

### 1. The Orchestration & State Engine

**Current State:** The orchestrator manages background jobs using in-memory `asyncio.Lock` dictionaries for both runs and Google Docs creation. State persistence relies on synchronous read-merge-write operations to local JSON files (e.g., `storage.write_nlm_state` ).

**The Architectural Upgrade: Agentic State Machine (LangGraph) & Distributed Locking**

* 
**Action (State Engine):** Transition the two-stage NotebookLM pipeline from a linear asynchronous function (`_run_sequence` ) into a formal agentic workflow using a state machine framework like LangGraph. Right now, if the server crashes during Stage 2, the lock vanishes, the state file is orphaned, and recovery requires manual intervention. A state graph natively provides checkpointing; if a node (e.g., "Draft 1 Generation" or "Scholarly Elaboration") fails, the graph can seamlessly resume from the last successful checkpoint rather than restarting or failing entirely.


* 
**Action (Locking):** Replace `_chapter_doc_locks` and `_run_locks`  with a distributed lock manager (like Redis). In-memory locks tie you to a single-server deployment. If you ever spin up multiple workers to handle batch processing concurrently, those workers will clobber each other's local locks, leading to race conditions in Google Docs creation or NotebookLM chat sessions.



### 2. The Ingestion Pipeline

**Current State:** The ingestion flow directly manages NotebookLM via browser automation bindings, enforcing a strict 50-source limit. It handles the vector database indexing delay by using a hardcoded `await asyncio.sleep(10)` buffer and wraps uploads in 3-minute timeout blocks.

**The Architectural Upgrade: Event-Driven Ingestion & Backend Abstraction**

* **Action (Indexing):** Eliminate arbitrary sleep timers. Hardcoded waits like `asyncio.sleep(10)`  are notoriously brittle—they waste time when indexing is fast and fail silently when indexing is slow, leading to LLM hallucinations because the RAG context wasn't ready. Implement a deterministic polling loop that queries the source status until it explicitly returns a "READY" state before allowing the prompt to fire.


* **Action (RAG Abstraction):** Decouple the business logic from NotebookLM. While NotebookLM is excellent, the 50-source limit  will eventually bottleneck large PhD dissertations. Abstract the ingestion layer behind a generic interface so that, in the future, you can seamlessly hot-swap NotebookLM for a custom RAG pipeline (using vector databases like Pinecone or Milvus) without rewriting `notebooklm_service.py`.



### 3. The File Management & Resolution Layer

**Current State:** File resolution attempts to match thesis names to pre-scanned local folder paths (`drive_scan_result`). Conflict detection in Google Docs relies on normalizing whitespace and comparing raw strings to detect manual edits.

**The Architectural Upgrade: Deterministic Sync & Content Hashing**

* **Action (File Resolution):** Move away from string-matching folder paths. Local directory structures change, causing paths to break. Instead, transition to a content-addressable storage model. When a user registers a thesis, generate a SHA-256 hash of the PDF contents and store that in your database alongside its Google Drive ID. The orchestrator should look up files by their hash or Drive ID, making the system immune to local file renaming or folder restructuring.


* 
**Action (Google Docs Sync):** Replace the binary text-matching conflict guard  with a granular diffing engine. Academic writing is highly iterative. If a user manually fixes a single comma in Google Docs, your current system raises a `GDocsConflictError` and refuses to sync. Instead, implement a lightweight Operational Transformation (OT) or Diff-Match-Patch algorithm. This allows the orchestrator to merge the AI's structural updates with the user's manual prose tweaks, drastically reducing friction.
